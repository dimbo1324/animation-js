# Рекомендации по производительности анимаций

Свод правил для написания быстрых DOM/JS-анимаций без библиотек. Применять при разработке любых сцен, персонажей и игровых циклов в этом проекте.

## 1. `requestAnimationFrame()` вместо `setInterval()`

Это стандарт для анимаций.

```js
function animate() {
  // обновляем состояние
  // рисуем

  requestAnimationFrame(animate);
}

requestAnimationFrame(animate);
```

Браузер сам подстроится под частоту монитора (60, 120, 144 Гц) и не будет выполнять лишние кадры.

## 2. Менять только `transform` и `opacity`

Самое быстрое (не вызывает layout/reflow):

```css
transform: translateX(100px);
transform: rotate(30deg);
opacity: 0.5;
```

Гораздо хуже:

```
left
top
width
height
margin
padding
```

Потому что они заставляют браузер пересчитывать расположение элементов.

## 3. Не читать и писать DOM вперемешку

Плохо:

```js
div.style.left = '100px';

const w = div.offsetWidth;
```

После изменения стиля браузер может сразу пересчитать layout.

Лучше:

```js
const w = div.offsetWidth;

// потом все изменения
div.style.transform = ...
```

## 4. Хранить состояние в объектах

Не так:

```js
const x = parseInt(div.style.left);
```

Лучше:

```js
const player = {
  x: 100,
  y: 200,
  vx: 3,
};
```

И только после вычислений применять:

```js
div.style.transform = `translate(${player.x}px, ${player.y}px)`;
```

DOM становится лишь отображением данных.

## 5. Разделить обновление и отрисовку

```
Update()
    ↓
Physics
    ↓
AI
    ↓
Animation
    ↓
Render()
```

Так устроены игры.

## 6. Использовать классы

Например:

```js
class Person {
  update(dt) {
    this.x += this.speed * dt;
  }

  render() {
    this.el.style.transform = `translate(${this.x}px, ${this.y}px)`;
  }
}
```

А потом:

```js
people.forEach((p) => p.update(dt));
people.forEach((p) => p.render());
```

## 7. Не создавать объекты каждый кадр

Плохо:

```js
function update() {
  const point = {
    x: 0,
    y: 0,
  };
}
```

Лучше менять существующие объекты.

## 8. Если объектов очень много — Canvas

Когда элементов становится:

- 500
- 1000
- 5000

DOM начинает проигрывать.

Тогда используют:

- Canvas 2D
- WebGL
- WebGPU

Там рисуется один большой холст, а не тысячи `<div>`.

## 9. Использовать дельту времени (`deltaTime`)

Не так:

```js
x += 5;
```

А так:

```js
x += speed * dt;
```

Тогда скорость не зависит от FPS.

## 10. Пространственные структуры

Если персонажей сотни, не нужно проверять каждого с каждым.

Вместо:

```
1000 × 1000 = 1 000 000 проверок
```

используют:

- Grid
- QuadTree
- Spatial Hash

И проверяют только соседей.

## Лестница уровней разработки

1. **HTML + CSS-анимации** — простые эффекты.
2. **DOM + JavaScript (`requestAnimationFrame`)** — интерактивные сцены с десятками объектов.
3. **Canvas 2D** — сотни и тысячи движущихся объектов.
4. **WebGL/WebGPU** — сложная графика, эффекты, полноценные игры и симуляции.

Рекомендация: сначала делать несколько проектов на DOM + `requestAnimationFrame()`. Переход на Canvas имеет смысл, когда упираешься в производительность или ограничения DOM — тогда он будет осознанным шагом, а не преждевременной оптимизацией.
